# HTTP API сервиса сбора логов (Log Collector)

## Общая информация

Сервис предоставляет HTTP API для приёма и выдачи логов. Логи сохраняются в PostgreSQL через объект `PGStorage`. Обработка входящих запросов на запись выполняется асинхронно в отдельных задачах, не блокируя HTTP-сессию.

**Технологии**: C++, Boost.Beast, Boost.JSON, Poco, PostgreSQL.

**Формат даты/времени**: в логах используется UTC время в формате `YYYY-MM-DD HH:MM:SS`.

**CORS**: для JSON-ответов добавляется заголовок `Access-Control-Allow-Origin: *`.

## Базовый URL

Все эндпоинты доступны по корневому пути сервера. Например: `http://<host>:<port>/`

## Эндпоинты

### 1. GET /

Возвращает статус сервиса и состояние подключения к RabbitMQ (симулируется наличием хранилища).

**Параметры запроса**: отсутствуют.

**Тело запроса**: отсутствует.

**Пример ответа** (200 OK):
```json
{
  "status": "online",
  "service": "Log Collector API",
  "rabbitmq": "connected"
}
```

**Возможные коды ответа**:
- `200 OK` – сервис работает.

---

### 2. GET /health

Простая проверка работоспособности. Возвращает статус, если хранилище доступно.

**Параметры запроса**: отсутствуют.

**Тело запроса**: отсутствует.

**Пример ответа** (200 OK):
```json
{
  "status": "online"
}
```

**Пример ответа при ошибке** (500 Internal Server Error):
```json
{
  "detail": "Internal server error"
}
```

---

### 3. POST /log

Отправка одного лог-сообщения. Данные передаются в теле запроса в формате JSON.

**Тело запроса** (JSON-объект):
```json
{
  "o": "owner_name",      // обязательное, строка (trim)
  "t": "token_value",     // обязательное, строка (trim)
  "m": "log message",     // обязательное, строка (trim)
  "type": "INFO",         // обязательное, строка (trim) (не сохраняется в БД)
  "dt": "2026-04-08 14:35:22,123"  // необязательное, если не указано – подставляется текущее время
}
```

**Валидация**:
- Поля `o`, `t`, `m`, `type` обязательны и не должны быть пустыми после trim.
- Формат `dt` (если указан) должен быть строкой, пробелы обрезаются.

**Пример ответа** (200 OK):
```json
{
  "status": "success"
}
```

**Обработка**:
- Сообщение помещается в фоновую очередь для записи в БД (метод `PGStorage::push`). Только поля `owner`, `token`, `message` передаются в хранилище. Поле `type` не используется при сохранении.

**Коды ответа**:
- `200 OK` – сообщение принято.
- `400 Bad Request` – некорректный JSON или нарушение валидации полей.
- `500 Internal Server Error` – внутренняя ошибка.

---

### 4. POST /log/batch

Пакетная отправка нескольких лог-сообщений.

**Тело запроса**: JSON-массив объектов, каждый соответствует формату из `POST /log`.

**Пример запроса**:
```json
[
  {
    "o": "app1",
    "t": "token1",
    "m": "First message",
    "type": "INFO"
  },
  {
    "o": "app2",
    "t": "token2",
    "m": "Second message",
    "type": "WARN",
    "dt": "2026-04-08 12:00:00,000"
  }
]
```

**Пример ответа** (200 OK):
```json
{
  "status": "success",
  "message": "2 logs accepted and queued",
  "count": 2
}
```

**Примечания**:
- Каждое валидное сообщение отправляется в фоновую очередь независимо.
- Невалидные элементы пропускаются (счётчик успешных увеличивается только для корректных).

**Коды ответа**:
- `200 OK` – даже если часть сообщений отклонена, но массив был обработан.
- `400 Bad Request` – некорректный JSON или корневой элемент не является массивом.

---

### 5. GET /log/pull

Возвращает список логов, начиная с указанного смещения (offset).

**Параметры запроса**:
- `offset` (query-параметр) – целое неотрицательное число. По умолчанию `0`.

**Пример запроса**: `GET /log/pull?offset=10`

**Пример ответа** (200 OK):
```json
{
  "logs": [
    {
      "offset": 11,
      "owner": "app1",
      "token": "token1",
      "dt": "2026-04-08 14:35:22",
      "msg": "Log message content"
    },
    {
      "offset": 12,
      "owner": "app2",
      "token": "token2",
      "dt": "2026-04-08 14:36:01",
      "msg": "Another log"
    }
  ]
}
```

**Примечание**: поле `dt` форматируется как `YYYY-MM-DD HH:MM:SS` (без миллисекунд).

**Коды ответа**:
- `200 OK`
- `500 Internal Server Error` – ошибка при запросе к БД (детали в поле `detail`).

---

### 6. GET /log/pull-plain

Аналогичен `/log/pull`, но возвращает данные в текстовом формате (plain text), удобном для чтения человеком или консольного просмотра.

**Параметры запроса**:
- `offset` – аналогично предыдущему.

**Пример ответа** (200 OK, `Content-Type: text/plain`):
```
11[2026-04-08 14:35:22]: Log message content
12[2026-04-08 14:36:01]: Another log
```

Если логов нет – возвращается строка `"No logs available"`.

**Заголовки**:
- `Cache-Control: no-cache, no-store, must-revalidate`
- `Pragma: no-cache`
- `Expires: 0`
- `Vary: Accept-Encoding`

---

### 7. GET /log/download-plain

Похож на `/log/pull-plain`, но дополнительно устанавливает заголовок `Content-Disposition: attachment` с именем файла, генерируемым по текущему UTC времени.

**Параметры запроса**:
- `offset` – аналогично.

**Имя файла**: `maintenance_YYYYMMDD_HHMMSS.txt`

**Пример ответа** (200 OK, `Content-Type: text/plain`):
```
11[2026-04-08 14:35:22]: Log message content
...
```

**Заголовки**:
- `Content-Disposition: attachment; filename="maintenance_20260408_143522.txt"`
- `Cache-Control: no-cache, no-store, must-revalidate`
- `Pragma: no-cache`
- `Expires: 0`
- `Vary: Accept-Encoding`

---

## Модели данных

### LogMessage (входная модель)
Поля после парсинга JSON:
- `owner` (string) – идентификатор владельца/источника лога.
- `token` (string) – токен (например, идентификатор устройства или сессии).
- `message` (string) – текст сообщения.
- `type` (string) – тип/уровень логирования (INFO, WARN, ERROR и т.п.).
- `datetime` (string) – временная метка в формате `YYYY-MM-DD HH:MM:SS,mmm`; если не указана, заполняется автоматически.

Метод `valid()` проверяет, что `owner`, `token`, `message`, `type` не пусты после удаления пробельных символов.

### MaintenanceLogRecord (выходная модель из БД)
Структура, возвращаемая методом `PGStorage::pull()`:
- `offset` (int) – порядковый номер записи в таблице логов.
- `owner` (string)
- `token` (string)
- `timestamputc` (Poco::DateTime) – UTC время записи.
- `msg` (string) – сообщение.

## Коды ответа HTTP

| Код  | Описание                                                                           |
|------|------------------------------------------------------------------------------------|
| 200  | Успешное выполнение запроса.                                                       |
| 400  | Некорректный запрос (невалидный JSON, отсутствуют обязательные поля, не массив).    |
| 404  | Запрошенный путь не существует.                                                     |
| 500  | Внутренняя ошибка сервера (проблемы с БД, исключения).                              |

При ошибках 400 и 500 в теле ответа (JSON) содержится поле `error` или `detail` с описанием.

---

## Асинхронная обработка записи

Для методов `POST /log` и `POST /log/batch` после валидации сообщения передаются в фоновый пул потоков через `boost::asio::post`. Это позволяет быстро отвечать клиенту, не дожидаясь фактической вставки в БД. Ошибки записи в БД логируются в `stderr`, но клиенту не возвращаются.

---

## Примечания

- Сервер использует один или несколько потоков (конфигурируется `num_threads`) для обработки HTTP-запросов и фоновых задач.
- Подключение к PostgreSQL проверяется при старте сервера вызовом `testConnection()`. В случае неудачи сервер не запускается.
- В логах сервера (std::cerr) выводятся сообщения о принятых логах и количестве активных соединений.
- Запросы к `/log/pull*` выполняются синхронно в том же потоке HTTP-сессии, поэтому при большом объёме данных возможна задержка ответа.
